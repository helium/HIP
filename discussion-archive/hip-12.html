cokes 01/05/2021 10:31 AM
HIP12 Remote Assert Amendment Wanted to provide an update on this HIP and explore an alternative way to approach the same problem but with a different solution. Today, a user would need physical access to the Hotspot because you need the Hotspot to partially sign the transaction and send it back to the app over bluetooth. Once the app has the partial transaction, it signs once again with the owner address and submits to the blockchain. This method makes it difficult for host/owners to set up their hotspots, as owners would need to pre-assert before shipping to hosts, and if the host moves, hotspots need to be returned to be reasserted. A costly operation. HIP 12 aims to solve this problem because it allows the owner to remotely the assert the Hotspot without being physically colocated with the Hotspot. This works particularly well for mining pools like Emrit, Zatafi, CoastFi and others, but outside of that use case, it isn't particularly meaningful to the network and relaxes the security requirement of asserting a location. We can look at solving this problem another way and keep security in mind. With the upcoming changes to the Hotspot app that allows anyone to follow a Hotspot and run diagnostics, update wi-fi, etc over bluetooth (similar to Ken's Hotspot Utility app), it's entirely possible that we expose Assert Locations in settings that allows a Host to construct a partial assert location transaction and send it to the Hotspot owner to approve and countersign, similar to how transfer Hotspot works today. (contd)
10:31
Example UX: * Host sees Hotspot in their app > taps Settings > pair with Hotspot over bluetooth * Host asserts location, submits partial transaction to Owner. * Owner receives notification of pending assert, chooses to accept or deny. * If owner accepts, the owner signs the transaction, submits to the blockchain, and HNT is deducted from owner account (for the transaction, and for assert if applicable). This would allow us to enforce that the action was initiated by someone local but confirmed by the hotspot owner. We already have this two party interaction modeled in the Transfer Hotspot interaction in the current app's UX. We could also imagine this being supported via other methods like a local HTTP server rather than BLE for future hotspots that won't have bluetooth. Further security improvements can take advantage of matching the Hotspot GPS coordinates with the device GPS location kind of like an "Assert Fingerprint". There is potential for spoofing here, but it's something to be solved in implementation. We also are aware some Hotspots will not have GPS, so the logic around this can be ironed out down the road. With this said, I'd like to gauge the community on rejecting HIP 12 and/or amending it with improved security as described above.

rawrmaan 01/06/2021 10:31 AM
As the author of this HIP I'll stand by the HIP as written. The only changes to the UX that I would propose are removing the need to connect to the hotspot, and adding an address resolution/reverse geolocation search field for remote assertions. I want everything to be as simple as possible for my hosts, and don't want them to have to deal with dropping a pin on the map, and potentially making errors doing so, leading to a time consuming back and forth. This is especially important for hosts who are not technically inclined. This is what I want my flow to look like as a patron: 1. Host signs up with me and provides their address 2. I generate a QR code on my web UI that contains a location assert with the lat/lng coords and hotspot address 3. I scan the QR code with my Helium app, just like I do now for sending payments to my hosts 4. I sign and send the transaction with one tap 5. I send my hotspot to the host. Now all they need to do is connect wi-fi or Ethernet. I think it adds unnecessary friction to require hosts to do something additional via the app, and this could affect the scalability of Patron operations, especially as they get really huge. I believe that the fewer steps the host has to complete, the better. I also don't think it has much of an effect on the security of the network to certify GPS via a mobile device. Especially since we have already decided that GPS isn't relevant to the network and won't be in future hotspots. The current anti-gaming measures are taking the worst possible scenarios into account anyway, i.e. where someone is going through the maximum possible effort to spoof their hotspot's location. This renders any GPS checking redundant. (edited)

HIP12 Remote Assert Amendment Wanted to provide an update on this HIP and explore an alternative way to approach the same problem but with a different solution. Today, a user would need physical access to the Hotspot because you need the Hotspot to partially sign the transaction and send it back to the app over bluetooth. Once the app has the partial transaction, it signs once again with the owner address and submits to the blockchain. This method makes it difficult for host/owners to set up their hotspots, as owners would need to pre-assert before shipping to hosts, and if the host moves, hotspots need to be returned to be reasserted. A costly operation. HIP 12 aims to solve this problem because it allows the owner to remotely the assert the Hotspot without being physically colocated with the Hotspot. This works particularly well for mining pools like Emrit, Zatafi, CoastFi and others, but outside of that use case, it isn't particularly meaningful to the network and relaxes the security requirement of asserting a location. We can look at solving this problem another way and keep security in mind. With the upcoming changes to the Hotspot app that allows anyone to follow a Hotspot and run diagnostics, update wi-fi, etc over bluetooth (similar to Ken's Hotspot Utility app), it's entirely possible that we expose Assert Locations in settings that allows a Host to construct a partial assert location transaction and send it to the Hotspot owner to approve and countersign, similar to how transfer Hotspot works today. (contd)
