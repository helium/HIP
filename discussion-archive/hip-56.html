# HIP 56: Improved State Channel Disputes - Author(s): macpie, michaeldjeffrey, @professor , @hashc0de , et al - Start Date: 2022-02-21 - Category: Technical - Original HIP PR: https://github.com/helium/HIP/pull/366 - Tracking Issue: https://github.com/helium/HIP/issues/369 - Status: In Discussion ## Summary This HIP describes an update to the dispute mechanism for State Channels that would improve blockchain scaling, better reconcile Data Transfer activity and rewards for Hotspots, and address recent issues around large block sizes. State Channel disputes provide a mechanism to validate the integrity of a State Channel. Previously, disputes triggered by participating Hotspots contained redundant information, and increased block size. On a couple of occasions the number of disputes caused some large blocks to halt chain activity due to Validators not being able to consume them. This proposal retains the State Channel integrity check by any participant, but also updates the dispute mechanism to prevent disputes from halting chain activity. ## Rendered View https://github.com/helium/HIP/blob/main/0056-state-channel-dispute-strategy.md (edited)

hashc0de 02/23/2022 1:41 PM
Two (maybe 3) potential things happen with an initial close: 1. State Channel Owner closes SC. It's signed by them. It contains summaries of all activity. This is the happy path and everyone should be happy. 2. Someone other than the Owner closes the State Channel. They close with the last version of the State Channel that they have which may be incomplete. They do this after the Grace Period expires for the Owner because they want to get paid. 3. (edge case): State Channel Owner somehow loses data and closes a SC with incomplete data. In the #2 (and 3) failure case above: A different State Channel Actor notices they weren't credited and they file an additional close transaction. There are a couple of potential scenarios: 1. If it's additive, no problem. The chain trusts the secondary close and has proof because the Owner signed the accepted offers. This is not considered a dispute. 2. State Channel Actor files a true dispute where the Hotspot has a different value than the State Channel Closer. This goes into dispute and this HIP is at play. The dispute would only affect the 2k actors in the channel as far as not getting paid for their traffic. The State Channel Owner still loses their DCs because of their Dispute. They'll do better next time. And if a Hotspot doesn't trust a Router in the future, they could, potentially, stop sending / trusting that Router for future traffic. There's another case where a State Channel Actor makes up a fake close but this would be an invalid transaction. The chain would reject this. Bottom line is that Routers failing to close State Channels correctly is rare and due to an edge case, but right now, that edge case can have disastrous effects on the chain. This HIP proposes a safety mechanism to avoid that and get back to rewarding Data Transfer correctly. (edited)
